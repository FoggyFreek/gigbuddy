# Subscriptions, entitlements & platform billing

Paid tiers (bronze/silver/gold) gate features and limits per tenant; billing runs on Mollie behind a provider port. Migrations `100`–`105`. The full rev-5 design lives in the plan file referenced by project memory; this skill is the code-level map and the invariants you must not break.

## Plans & entitlements

- **`shared/entitlements.js` is the single source of truth** (server re-export: `server/auth/entitlements.js`; frontend mirror types in `src/auth/entitlements.ts`). Features: `finance`, `integrations`, `customization`, `song_files`, `chordpro`, `public_promotion`. Limits: `storage_mb`, `members`, `bands` — **`null` = unlimited**. Plans store *complete* entitlement objects (every key present, `validateEntitlements`); `mergeEntitlements` overlays per-subscription `entitlement_overrides` and silently ignores malformed keys.
- Plan catalog: `subscription_plans` (migration `100`), default tiers seeded from `server/db/defaultPlans.js` **and** the migration SQL — keep both in sync. Bronze is `is_fallback` (free, always active, 0-priced, undeletable/unrenamable). Price semantics: `NULL` = interval unavailable, `0` = free-fallback only, `>0` = paid. Super-admin CRUD at `/admin/plans`; a subscription snapshots `price_cents` at subscribe/change time, so plan price edits never affect running subscriptions.
- Tier logos `public/icons/gb_{bronze,silver,gold}.png` are keyed **by plan slug** via `src/utils/planLogo.ts` — custom slugs return `null` and callers fall back to the standard app logo. The AppShell header logo swaps to the tier logo only when not locked/unenforced.

## Ownership: subscriptions are user-level, tenants inherit

- A subscription belongs to a **user** (one live non-canceled row per user, partial unique index). A tenant's entitlements come from `tenants.owner_user_id`'s subscription. **`owner_user_id IS NULL` (legacy tenant) skips enforcement entirely** — every gate passes, the frontend hides nothing. Deliberately no backfill; super admins migrate legacy tenants by PATCHing `owner_user_id` (`/admin/tenants`).
- Self-service creation (`tenantSelfService.js`, `/api/tenants`): the creator becomes owner + `tenant_admin`. The **bands limit** caps *active owned* tenants — checked under a `FOR UPDATE` user-row lock in the same transaction as insert/unarchive (archiving parks a band; unarchive re-checks the cap so it's not a loophole). Super-admin creation in `tenantService.js` stays uncapped/ownerless.

## Resolution & gating

- `entitlementService.resolveTenantEntitlements` **enforces all time bounds itself on read** — trial end + 2d grace, period end + 2d grace (+7d max while a renewal charge is still nonterminal at Mollie/SEPA), past_due + 14d. The scheduler only repairs durable status; **access never depends on it running**. Locked = fall back to the fallback plan's entitlements (fallback-lock; data is never deleted on lapse). `financeReadOnly` = plan lacks finance but the tenant has finance data.
- Server gates (`server/middleware/entitlements.js`, mounted after `requireApproved` → `resolveTenantId`): `requireEntitlement(feature)` denies 403 `{ code: 'entitlement_required', feature }`; `requireEntitlementForWrites(FEATURES.FINANCE)` is the finance read-only mode (GET/HEAD/OPTIONS pass — a downgrade must not cut users off from their own records; the fiscal retention duty is theirs). Credential/token GET+DELETE endpoints stay ungated — a downgraded admin must always be able to see and erase secrets.
- Numeric caps go through `limitService.enforceMemberCap/enforceBandCap` — **must run inside the caller's transaction** (the `FOR UPDATE` locks only protect check-then-insert when the insert commits in the same txn); 409 `{ code, limit }`. Storage is reservation-based (`storageService.uploadObjectWithQuota` is THE upload entry point → 413 `StorageQuotaError`); failed uploads land in `storage_cleanup_queue`, drained by the reconciliation tick.
- Frontend: `/auth/me` carries `entitlements` (`flags`/`limits`/`planSlug`/`locked`/`financeReadOnly`; `null` = ownerless → allow everything). `useEntitlements()` mirrors `usePermissions`; `RequireEntitlement` is the route guard. **Tier-locked nav items stay visible** with a `DiamondOutlined` icon linking to `/upgrade/:feature` (`LockedFeaturePage` upsell) — role-gated items are hidden instead. All of this is presentation; the API gate is the defense.

## Billing flows (Mollie behind a port)

- **Provider port**: `server/billing/paymentProvider/` — code imports `getPaymentProvider()` (selected by `BILLING_PROVIDER`, key `PLATFORM_MOLLIE_API_KEY`), never a concrete adapter, and speaks **canonical statuses only** (`statuses.js`); Mollie types stay inside `mollieProvider.js`. Tests inject a fake via `setPaymentProviderForTests` (fake class in `src/tests/server/_fakeProvider.js`; subscription-row helpers in `_billing.js`).
- **Local state first, remote second, never a remote call inside a DB transaction.** Subscribe commits a `pending_mandate` row, then creates a €0.01 mandate-establishing checkout; an abandoned signup is just a stale row the scheduler cancels after 24h. 7-day trial, once per user (`trialing` after the mandate pays; otherwise `pending_activation` — **no access until an authoritatively-paid charge**).
- **Saga/outbox**: every remote mutation goes through `billingSaga.js` — claim a `billing_operations` row (deterministic idempotency key, `billingShared.idemKeys`) *before* the provider call, mark `succeeded`/`failed_retryable`/`failed_terminal` after. A resumed saga finds the succeeded row and skips the call. The provider schedule is created/replaced by `repairSchedule`, keyed off `mollie_schedule_stale`; terminal repair failures flag `billing_repair_needed` (visible in `/admin/subscriptions`).
- **Ingestion is one funnel**: webhook (`/api/public/billing/mollie/webhook` — always 200, posted id is a routing hint only, status re-fetched from the provider, payment's customer verified against the owner) and the reconcile poll both call `ingestProviderPayment` → `applyPaymentOutcome` (single txn under the subscription row lock). The status-transition predicate lives **in SQL** (`billing_payment_transition_allowed`, migration `104`) inside the upsert's `ON CONFLICT ... WHERE`, so concurrent webhook/reconcile replays are race-free and inert transitions never re-fire effects. Billing notifications are user-level (`tenant_id` nullable, migration `103`) and inserted in the same transaction with a `dedupe_key`.
- **Plan changes are activate-first**: upgrade/interval sets the `pending_*` columns, then an on-demand mandate charge; the *paid* event switches plan + period immediately and defers schedule repair. A failed charge drops the pending change. Trial changes switch free (no charge). Downgrades are implemented (migration `105`) — see the dedicated section below.
- Cancel with a paid period → `cancel_at_period_end`; otherwise immediate. Resume clears the flag and recreates the schedule. Complimentary subscriptions (super-admin grant at `/admin/subscriptions`) carry no provider objects, are excluded from Mollie-touching tasks, and block user self-management (`complimentary_managed_by_admin`).
- **Scheduler**: `runReconciliationTick` (15 min, single-instance via `pg_try_advisory_lock('billing_reconciliation')` on one checked-out client) runs the repair-only `BILLING_TASKS` (stale signups, nonterminal-payment polling, schedule repairs, downgrade-saga resume + replacement poll, pending-downgrade period-end flip, past-due force-cancel, cancel-at-period-end finalize, pending-purge safety net, trial reminders, orphaned ops, expired complimentary) plus the storage-cleanup drain. Mollie does its own payment retries — **no custom dunning**. Off in tests / `BILLING_SCHEDULER_DISABLED`.
- Local dev: `MOLLIE_DISABLE_WEBHOOK=true` omits webhook URLs; `POST /api/billing/sync` (the UI sync button) re-ingests nonterminal payments manually. `/api/billing` is user-level (`requireApproved`, no tenant resolution).

## Downgrade & purge (phase 6, migration `105`)

Downgrades are the one flow that deletes data, so every branch runs on **informed consent first, no data loss before the target plan is real**.

- **Confirm** (`billingService.downgrade`, type-to-confirm phrase `downgrade to <slug>`, validated server-side): the confirm transaction freezes a **purge manifest** (`featuresToPurge` over *effective* entitlements — per-subscription overrides survive, so an override-granted feature is never purged) and a **limits snapshot** (`pending_limits_snapshot`) that binds capacity growth immediately via the resolver (`min(current, target)`). Three branches: **trial** → switch/cancel immediately + purge right after commit; **free fallback** → rides `cancel_at_period_end`, purge runs when the period-end finalize task cancels; **paid lower** → `pending_change_kind='downgrade'` + replacement-subscription saga.
- **Blockers**: `computeDowngradeBlockers` checks member/storage limits against **ALL owned tenants, archived included** (an archived band can be unarchived onto the lower plan — enforcing at unarchive instead would deadlock, since archived tenants 403 all writes and could never be trimmed); only the **band cap** counts active tenants (archiving is the documented way to satisfy it). The confirm re-runs it under the exact lock set every capacity-growing write takes (user row, then tenant rows id-ascending + advisory) → 409 `over_target_limit`.
- **Paid-lower saga** (`scheduleDowngradeReplacement`): cancel the OLD provider subscription (the immutable `superseded_mollie_subscription_id` captured at confirmation — a resumed saga can never cancel the replacement) → create the replacement (first charge at period end; idempotency key on the *stable* period end so a retry crossing the boundary can't create a second one; **metadata `billingMetadata(sub.id, 'downgrade')` stamped at creation**) → guarded atomic repoint (`applyDowngradeSchedule`: only `WHERE downgrade_schedule_pending = TRUE AND status <> 'canceled'`). **A lost guard means a cancel/finalize/activation won the race** — the saga then cancels the now-orphaned replacement remotely unless the row already owns it (activation repointed first), flagging `billing_repair_needed` if even that fails. `downgrade_schedule_pending` is the durable resume marker.
- **Activation — only the replacement's charge activates**: the superseded id is always rejected; the replacement is recognized by the local repoint **or, when its first charge beats the repoint, by the create-time metadata** (`ingestProviderPayment` fetches the charge's provider subscription and passes a verified `replacementSubscriptionId` hint; Mollie echoes create-time metadata on fetch). `applyDowngradeActivation` switches plan/period, repoints `mollie_subscription_id` itself if needed, and **keeps the manifest** for the post-commit purge. A paid charge of unknown provenance whose attributing lookup failed is **never recorded** — a recorded paid outcome is permanently inert under the SQL transition predicate, so ingestion throws (retryable) instead and a later ingest starts fresh.
- **Period end without payment**: `reconcilePendingDowngrades` flips to `pending_activation` (resolver fallback-locks; nothing purged). A failed replacement charge is NOT terminal (Mollie retries ~daily); terminal = 7d stale in `pending_activation` or the provider reports the replacement canceled → `finalizeFailedDowngrade` cancels both remote subscriptions, clears all downgrade state, and **purges NOTHING** (the customer never got the lower tier). A user cancel mid-pending likewise clears everything unpurged.
- **`executeDowngradePurge`**: idempotent and self-serializing (per-subscription session advisory lock; inline post-activation hook vs. `reconcilePendingPurges` safety net), consumes the manifest on success. Scope is **recovery-safe**: only manifest features still off on the *current* effective entitlements are purged — an admin plan edit after confirmation can only SHRINK the purge. Covers **ALL owned tenants including archived** (the promised deletion must survive unarchive). Finance is never purgeable. The integrations purge deletes credentials/secrets, except the tenant Mollie key with paid payment links: retained encrypted + hidden (`mollie_api_key_retained_at`, readable only via `loadRetainedIntegrationCredential`), deleted outright when no links remain.
- **Frontend**: `DowngradeDialog` is type-to-confirm and requires a *successfully loaded* preview (features to purge + blockers) before confirm enables — the server re-checks blockers anyway, but never let a user confirm a purge they haven't seen.
